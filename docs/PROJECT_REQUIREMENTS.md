# Project Requirements — Spend Analyzer

## 1. Project Intent

Spend Analyzer is a learning-first, production-style backend project.

The product problem is personal finance analysis from bank and credit card statements. The engineering goal is to learn and apply:

1. Python backend development
2. FastAPI service design
3. PostgreSQL-backed data modelling
4. PDF ingestion and parsing
5. AI-assisted fallback parsing
6. Agentic AI workflows
7. Retrieval-Augmented Generation pipelines
8. Production-style modular architecture
9. Testing, validation, security, and observability

The project should not be optimized only for fastest feature delivery. It should be structured so each task teaches an important backend or AI engineering concept.

## 2. Primary Learning Objectives

### 2.1 Python

The project should teach practical Python through real backend modules, not isolated syntax exercises.

Learning goals:

- Python project structure
- Type hints
- Pydantic models
- Dependency management
- FastAPI routing and dependencies
- Service/repository layering
- File handling
- Error handling
- Unit testing with pytest
- Integration testing
- Working with PostgreSQL
- Clean code and refactoring

### 2.2 AI Engineering

The project should teach AI usage in a controlled backend system.

Learning goals:

- AI provider abstraction
- Prompt design
- Structured outputs
- AI fallback logic
- Confidence scoring
- Output validation
- Guardrails
- Cost-aware AI usage
- Evaluation datasets
- Regression testing for AI behavior

### 2.3 Agentic AI

The project should eventually include agentic workflows, but only after deterministic APIs are stable.

Learning goals:

- Tool selection
- Planning vs execution separation
- Agent state
- Safe tool invocation
- Query routing

Learning goals:

- Chunking
- Embeddings
- Vector storage
- Retrieval
- Re-ranking if needed

- Retrieval evaluation

## 3. Product Requirements

### 3.1 User Authentication

The system must support authenticated users.

Requirements:

- Use OAuth2/OIDC for login and token validation.
- Derive `user_id` from the validated JWT.
- Never trust client-supplied `user_id` for ownership.
- Every protected endpoint must enforce authentication.

### 3.2 Statement Upload

Users must be able to upload statement PDFs.

Requirements:

- Accept PDF uploads through `POST /ingest`.
- Reject non-PDF files.
- Store uploaded files safely.
- Store statement metadata.
- Track ingestion status.

### 3.3 PDF Extraction

The system must extract text and tables from uploaded PDFs.

Requirements:

- Use deterministic extraction first.
- Handle corrupt, empty, unsupported, and password-protected PDFs safely.
- Do not log full statement contents.

### 3.4 Transaction Parsing

The system must parse extracted statement content into normalized transactions.

Required parsing flow:

```text
Generic Parser → Broad Bank/Account Parser → AI Fallback → Manual Review
```

Requirements:

- Avoid one parser per card product.
- Prefer broad parser classes by bank and account type.
- Validate parsed transactions before persistence.
- Store parser confidence and parser run metadata.
- Mark uncertain statements as `NEEDS_REVIEW`.

### 3.5 Transaction Storage

Validated transactions must be stored in PostgreSQL.

Requirements:

- Store normalized transaction records.
- Link transactions to statement and user.
- Support category, merchant, date, and monthly analysis.
- Enforce user isolation in all queries.

### 3.6 Analytics

The system must provide deterministic analytics.

Requirements:

- Monthly summary
- Category breakdown
- Merchant analysis
- Month-on-month comparison
- Basic anomaly detection

Important rule:

```text
SQL/backend logic calculates. AI explains.
```

### 3.7 AI Insights

The system should generate AI-assisted explanations from deterministic summaries.

Requirements:

- AI must not calculate financial totals.
- AI receives structured summary/comparison data.
- AI returns concise insights.
- No-data scenarios must be handled cleanly.

### 3.8 Natural Language Query

Users should be able to ask finance questions in plain English.

Requirements:

- Classify query intent.
- Map supported intents to safe backend operations.
- Never execute raw SQL generated by the user or AI.
- Generate readable answers from trusted backend results.

### 3.9 RAG

The system should support semantic retrieval over statement text and transaction context.

Requirements:

- Chunk statement text.
- Generate embeddings.
- Store vectors with user ownership metadata.
- Retrieve only user-owned chunks.
- Combine SQL and retrieved context for grounded answers.

### 3.10 Agentic Workflow

The system should support controlled agentic workflows after core query APIs are stable.

Requirements:

- Agent must choose from predefined tools only.
- Tools must be safe backend functions, not arbitrary SQL.
- Agent decisions must be logged.
- Unsupported requests must fail safely.
- Financial numbers must still come from SQL/backend logic.

## 4. Non-Functional Requirements

### Security

- No hardcoded secrets.
- No full statement contents in logs.
- JWT validation on protected APIs.
- User-level data isolation everywhere.
- Parameterized SQL only.

### Accuracy

- AI must not be source of truth for money calculations.
- All parsed transactions must be validated.
- Low-confidence results require review.

### Maintainability

- Modular service/repository design.
- Small independently testable modules.
- Broad reusable parsers instead of parser sprawl.
- Clear documentation per major subsystem.

### Learning Quality

Each major task should include:

- What is being built
- What Python concept is being learned
- What backend/AI concept is being learned
- Acceptance criteria
- Test expectations

---

## 5. Out of Scope for Early MVPs

The early MVPs should not focus on:

- Full production cloud deployment
- Complex frontend polish
- Full mobile app
- Auto-ingestion from every email/SMS format
- Advanced budgeting workflows
- Fully autonomous financial agents

These can be added after Python, ingestion, analytics, AI fallback, RAG, and controlled agent workflows are stable.
